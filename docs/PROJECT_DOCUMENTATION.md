# Документация проекта "Весенний_код_40"

`Весенний_код_40` - это MAX-бот для записи абитуриентов и студентов на мероприятия вузов.

## 1. Что умеет проект

- Показывает каталог вузов и мероприятий.
- Позволяет смотреть мероприятия по выбранному вузу или общий список по всем вузам.
- Ведет пошаговую регистрацию на мероприятие через кнопки.
- Защищает API-шлюз JWT-аутентификацией.
- Поддерживает привязку логина внешнего клиента к MAX ID через MAX-бота.
- Предоставляет браузерную SPA-панель управления для техадминов, админов и организаторов.
- Поддерживает выбор даты календарем на 6 месяцев вперед.
- Поддерживает выбор времени кнопками: часы `00`-`23`, минуты с шагом 5 минут.
- Позволяет вручную вводить дату и слоты, если пользователь не хочет пользоваться кнопками.
- Создает код записи и хранит регистрацию в PostgreSQL.
- Показывает пользователю только активные личные записи.
- Позволяет отменять запись.
- Разделяет личные записи и мероприятия, которыми пользователь управляет как организатор или админ.
- Поддерживает роли `applicant`, `organizer`, `admin`, `tech_admin`.
- Поддерживает регистраторов, назначенных на конкретное мероприятие.
- Поддерживает мультивузовую модель: админ и организатор привязаны к конкретному вузу.
- Позволяет организаторам и админам создавать, изменять, мягко удалять и восстанавливать мероприятия своего вуза.
- Отправляет уведомления участникам через сервис фоновых задач.
- Пишет технические и бизнес-логи в JSON.
- Экспортирует логи через Fluent Bit в Loki и показывает их в Grafana.

## 5. Пользовательские сценарии

### 5.1 Абитуриент или студент

Обычный пользователь открывает бота, принимает согласие на обработку минимальных данных профиля и попадает в главное меню.

Доступные действия:

- выбрать вуз;
- посмотреть мероприятия выбранного вуза;
- посмотреть мероприятия всех вузов;
- открыть карточку мероприятия;
- выбрать слот;
- подтвердить запись;
- получить код записи;
- посмотреть свои активные записи;
- отменить свою запись.

После успешной регистрации бот показывает кнопку возврата в главное меню.

### 5.2 Организатор

Организатор привязан к одному вузу. Он работает только с мероприятиями своего вуза.

Доступные действия:

- открыть меню организатора;
- посмотреть мероприятия организатора;
- создать мероприятие;
- изменить мероприятие;
- удалить мероприятие;
- восстановить мероприятие;
- посмотреть записи участников по мероприятию;
- найти запись по коду;
- менять статусы записей;
- отправлять уведомления участникам мероприятия.

Личные записи организатора на мероприятия других людей находятся отдельно в меню "Мои записи на мероприятия".

### 5.3 Админ вуза

Админ привязан к одному вузу и в рамках этого вуза является полноправным управляющим.

Доступные действия:

- все действия организатора по мероприятиям своего вуза;
- назначать организаторов своего вуза;
- снимать роли организаторов своего вуза;
- смотреть пользователей, относящихся к своему вузу;
- управлять мероприятиями своего вуза.

Админ не может управлять мероприятиями и пользователями чужого вуза.

### 5.4 Технический админ

Технический админ управляет всей системой.

Доступные действия:

- назначать админов;
- назначать организаторов;
- снимать роли;
- менять вуз у админов и организаторов;
- смотреть карточку пользователя;
- управлять ролями независимо от вуза.

Первого техадмина проще всего задать через `.env`:

```env
TECH_ADMIN_MAX_IDS=ваш_max_id
```

### 5.5 Регистратор мероприятия

Регистратор назначается на конкретное мероприятие организатором или админом этого вуза. Это не глобальная роль пользователя: один и тот же человек может быть регистратором на одном мероприятии и обычным участником на другом.

Доступные действия:

- открыть закрепленное мероприятие;
- посмотреть список участников;
- найти участника по коду записи;
- отметить участника как пришедшего.

Регистратор не может изменять мероприятие, удалять его, рассылать уведомления или менять статусы записей кроме отметки посещения.

## 6. Роли и права

| Роль | Где действует | Основные права |
| --- | --- | --- |
| `applicant` | Вся публичная часть | Каталог вузов и мероприятий, запись, отмена записи, просмотр своих активных записей |
| `organizer` | Только свой вуз | Создание, изменение, удаление и восстановление мероприятий своего вуза, работа с участниками |
| `admin` | Только свой вуз | Все права организатора плюс назначение организаторов своего вуза |
| `tech_admin` | Все вузы | Назначение админов и организаторов, смена вуза, снятие ролей, глобальное управление |
| Регистратор мероприятия | Только назначенное мероприятие | Список участников, поиск по коду, отметка посещения |

Роль и привязка к вузу хранятся в PostgreSQL. Если назначить роль человеку, который еще не писал боту, DataService создаст пользователя-заглушку. Когда человек напишет боту, его имя обновится, а роль сохранится.

## 8. Мероприятия, даты и слоты

Мероприятие состоит из общей карточки и набора слотов.

Основные поля:

- вуз;
- название;
- описание;
- дата;
- формат;
- место или ссылка;
- длительность слота;
- список временных слотов;
- лимит мест;
- статус удаления.

Дата выбирается календарем. Календарь показывает дни по дням недели и позволяет переключаться вперед в пределах 6 месяцев от текущего месяца.

Ручной ввод даты тоже поддерживается. Пользователь может ввести:

```text
05.06.2026
5.6.2026
05/06/2026
```

Внутри системы дата нормализуется к формату:

```text
ДД.ММ.ГГГГ
```

Слоты можно создать кнопками:

1. Выбрать час начала от `00` до `23`.
2. Выбрать минуты с шагом 5 минут.
3. Получить конечное время по длительности слота.
4. Добавить следующий слот или завершить создание.

Ручной ввод слотов тоже поддерживается:

```text
15:00-16:00, 17:00-18:00
```

Ограничения:

- конец слота должен быть позже начала;
- слоты не должны пересекаться;
- время должно быть валидным;
- дата должна попадать в допустимый диапазон;
- при ошибке бот просит ввести данные заново;
- на каждом пошаговом сценарии должна быть кнопка отмены.

Если мероприятие изменяется, активным участникам отправляется уведомление через сервис задач.

## 9. Записи участников

Пользователь записывается на конкретный слот мероприятия. После подтверждения создается запись с кодом.

В личном разделе "Мои записи на мероприятия" отображаются только активные записи. Отмененные записи там не должны висеть.

Организатор и админ видят записи в контексте управляемых мероприятий. Это отдельная зона интерфейса: она не смешивается с личными записями пользователя.

Для учета посещаемости в записи есть поля `attendedAt` и `attendedBy`. Они заполняются, когда организатор, админ или назначенный регистратор отмечает участника как пришедшего.

Типовая логика:

- пользователь выбирает мероприятие;
- выбирает слот;
- подтверждает запись;
- получает код;
- при необходимости отменяет запись;
- организатор может найти запись по коду и поменять статус.

## 10. Мягкое удаление мероприятий

Удаление мероприятия мягкое. Это значит, что запись в БД не уничтожается, а получает признак удаления.

Зачем так сделано:

- не теряются участники;
- не теряется история;
- мероприятие можно восстановить;
- публичный каталог не показывает удаленные события.

Админ и организатор своего вуза могут удалить и восстановить мероприятие. Технический админ может управлять правами и вузами, а доступ к мероприятиям должен проходить через правила роли и вуза.

### 11.1 Bot

Папка:

```text
src/bot
```

Задачи:

- принимает сообщения и callback-кнопки из MAX;
- хранит состояние пошаговых сценариев в runtime-памяти;
- показывает меню, карточки и пагинацию;
- удаляет или редактирует старые сообщения там, где это возможно;
- обращается к API gateway через `api-client.ts`;
- пишет технические и бизнес-логи через Pino.

### 11.2 API Gateway

Папка:

```text
src/api-gateway
```

Стек:

- TypeScript;
- Fastify.

Задачи:

- принимает запросы от бота, SPA-панели и внешних интеграций;
- скрывает внутренние адреса сервисов;
- проксирует доменные запросы в DataService;
- проксирует задачи уведомлений в TaskService;
- предоставляет `/health`.

### 11.3 DataService

Папка:

```text
src/data-service
```

Стек:

- TypeScript;
- Fastify;
- Drizzle ORM;
- Drizzle Kit;
- PostgreSQL;
- `pg`.

Задачи:

- хранит пользователей;
- хранит вузы;
- хранит мероприятия и слоты;
- хранит записи участников;
- считает свободные места;
- проверяет активные записи;
- управляет ролями и привязкой к вузу;
- применяет Drizzle-миграции при старте;
- выполняет bootstrap начальных данных для локального запуска.

### 11.4 TaskService

Папка:

```text
src/task-service
```

Стек:

- TypeScript;
- Fastify;
- BullMQ;
- Redis.

Задачи:

- принимает задачи уведомлений;
- кладет задачи в Redis-очередь;
- выполняет отправку уведомлений через MAX API;
- пишет технические логи worker-а.

### 11.5 Shared

Папка:

```text
src/shared
```

Задачи:

- хранит общие TypeScript-типы;
- хранит доменные функции;
- не зависит от бота, Fastify, PostgreSQL и Redis.

Это слой контрактов, который помогает сервисам говорить на одном языке.

## 12. База данных

Основная БД - PostgreSQL.

Ключевые таблицы:

| Таблица | Назначение |
| --- | --- |
| `universities` | Вузы |
| `users` | Пользователи MAX, роли, привязка к вузу |
| `events` | Мероприятия |
| `event_slots` | Временные слоты мероприятий |
| `registrations` | Записи пользователей на слоты |
| `event_registrars` | Назначения регистраторов на конкретные мероприятия |
| `external_logins` | Привязка логинов внешних клиентов к MAX ID и временные коды входа |

Схема таблиц описана в `src/data-service/schema.ts`. SQL-миграции генерируются Drizzle Kit и хранятся в `drizzle/`. При старте DataService применяет эти миграции через `drizzle-orm/node-postgres/migrator`, после чего выполняет bootstrap начальных данных для локального стенда.

Основные команды:

```bash
npm run db:generate
npm run db:migrate
```

`db:generate` создает новую SQL-миграцию по изменениям в `schema.ts`. `db:migrate` применяет миграции к базе из `DATABASE_URL`.

Данные PostgreSQL в Docker сохраняются в volume:

```text
springcat-postgres
```

## 14. Аутентификация

API gateway проверяет JWT перед проксированием запросов к внутренним сервисам. Исключения: `/health`, `/auth/login/start`, `/auth/login/verify`.

Bot -> API gateway:

- бот перед каждым запросом выпускает короткоживущий JWT;
- алгоритм подписи: `RS256`;
- приватный ключ находится только у бота;
- gateway проверяет токен публичным ключом;
- `user_id` берется из payload токена и прокидывается внутрь через `x-auth-user-id`;
- рекомендуемый TTL токена: 60 секунд.

Внешние клиенты -> API gateway:

- gateway выдает клиентский JWT после проверки одноразового кода;
- алгоритм подписи: `HS256`;
- симметричный секрет задается через `CLIENT_JWT_SECRET`;
- токен живет 15 минут.

Переменные окружения:

```env
BOT_JWT_PRIVATE_KEY_BASE64=...
BOT_JWT_PUBLIC_KEY_BASE64=...
CLIENT_JWT_SECRET=...
BOT_DEEPLINK_BASE=https://max.ru/spring_cat40_bot
```

Первичная привязка внешнего клиента:

1. Пользователь вводит логин во внешнем клиенте.
2. Клиент вызывает `POST /auth/login/start`.
3. Gateway возвращает deeplink и payload для QR.
4. Пользователь открывает ссылку в MAX или сканирует QR.
5. Бот получает `/start login_<login>` или `/login <login>`.
6. Если согласие не принято, бот сначала проводит пользователя через согласие.
7. Бот выпускает одноразовый код и отправляет его пользователю в MAX.
8. Пользователь вводит код во внешнем клиенте.
9. Клиент вызывает `POST /auth/login/verify`.
10. Gateway возвращает клиентский JWT.

Повторный вход:

1. Пользователь вводит логин во внешнем клиенте.
2. Gateway находит привязанный MAX ID и отправляет код в MAX через TaskService.
3. Пользователь вводит код во внешнем клиенте.
4. Gateway выдает новый клиентский JWT.

## 15. Логирование и наблюдаемость

Все сервисы пишут структурированные JSON-логи в stdout/stderr.

```env
LOG_LEVEL=info
```

Типовые значения:

- `debug` - подробная диагностика;
- `info` - обычный рабочий режим;
- `error` - только ошибки.

### 15.1 Типы логов

В логах используется поле:

```json
{
  "log_type": "technical"
}
```

или:

```json

}
```

`technical` - технические события:

- старт сервиса;
- HTTP-запрос;
- healthcheck;
- ошибка;
- событие worker-а;
- подключение к внешнему сервису.

`business` - события бизнес-процесса:

- создание мероприятия;
- изменение мероприятия;
- мягкое удаление мероприятия;
- восстановление мероприятия;
- регистрация пользователя;
- отмена регистрации;
- смена роли;
- постановка уведомления в очередь.

### 15.2 Pipeline логов

```text
service stdout JSON
  └──► Docker logs
        └──► Fluent Bit
              └──► Loki
                    └──► Grafana
```

Адреса:

- Grafana: `http://localhost:3000`;
- Loki: `http://localhost:3100`.

Логин и пароль Grafana по умолчанию:

### 17.2 Архитектурные подходы

Используется сервисное разделение по ответственности:

- бот отвечает за диалоги;
- gateway отвечает за входную точку;
- data-service отвечает за данные;
- task-service отвечает за фоновые задачи;
- shared отвечает за общие контракты.

Используется Gateway pattern:

- клиенты не знают внутренние адреса сервисов;
- SPA-панель и внешние клиенты обращаются в один backend endpoint;
- внутреннюю структуру сервисов можно менять без переписывания клиента.

Используется soft delete:

- удаленные мероприятия скрываются из каталога;
- записи участников не теряются;
- мероприятие можно восстановить.

Используется role-based access control:

- права зависят от роли;
- для админа и организатора важна привязка к вузу;
- техадмин имеет глобальную область действия.

Используется structured logging:

- логи являются JSON;
- технические и бизнес-события разделены;
- логи можно парсить и анализировать в Loki/Grafana.

Используется очередь задач:

- отправка уведомлений не блокирует основной пользовательский сценарий;
- задачи можно повторять при ошибках;
- worker можно масштабировать отдельно.

## 19. SPA-панель управления

Проект включает отдельную SPA-панель управления в директории `src/admin-spa`.

Схема работы:

```text
React + TS admin
  └──► api-gateway
        ├──► data-service
        └──► task-service
```

Панель написана на React + TypeScript, собирается Vite и использует официальный пакет компонентов `@maxhub/max-ui`.
Логин выполняется через реализованный flow одноразового кода и deeplink из MAX:

1. Пользователь открывает SPA-панель.
2. Вводит логин.
3. Если логин еще не привязан, панель показывает QR и deeplink в бота.
4. Пользователь открывает deeplink или сканирует QR.
5. Бот при необходимости проводит согласие, привязывает логин к MAX ID и отправляет код.
6. При повторном входе код сразу отправляется сообщением в MAX.
7. Пользователь вводит код в SPA.
8. Gateway проверяет код через DataService и выдает короткоживущий web JWT.

Если приложение MAX открылось по QR/deeplink, но не передало payload в чат, пользователь может написать боту вручную:

```text
/login ваш_логин
```

Внутрь панели допускаются только пользователи с ролями `organizer`, `admin`, `tech_admin`. Проверка есть на frontend для понятного сообщения пользователю и на API gateway для защиты операций.

Такой вариант сохраняет MAX как источник идентичности и не заставляет заводить отдельные пароли.

В панели доступны:

- управление мероприятиями роли организатора;
- расширенное управление мероприятиями роли админа в рамках своего вуза;
- глобальный просмотр и назначение ролей для техадмина;
- назначение и снятие регистраторов мероприятия;
- просмотр участников и отметка посещаемости по коду записи.

## 20. Безопасность

В проекте нельзя коммитить:

- `.env`;
- полный токен MAX-бота;
- приватный ключ `BOT_JWT_PRIVATE_KEY_BASE64`;
- `CLIENT_JWT_SECRET`;
- production-пароли;
- дампы БД с персональными данными.

Минимальные персональные данные:

- MAX ID;
- отображаемое имя;
- роль;
- привязка к вузу;
- записи на мероприятия.

Согласие пользователя фиксируется через `LEGAL_DOC_VERSION`. При изменении текста согласия версию нужно менять, чтобы можно было отличать старые и новые принятия.

Для production стоит дополнительно сделать:

- HTTPS перед API gateway;
- закрыть прямой внешний доступ к DataService и TaskService;
- заменить стандартный пароль Grafana;
- вынести PostgreSQL в managed DB или отдельный защищенный сервер;
- настроить backup PostgreSQL;
- ограничить доступ к Loki и Grafana;
- добавить rate limiting на gateway;
- добавить refresh-токены или серверные web-сессии для долгих браузерных сессий.

## 21. Масштабирование

Вертикальное масштабирование:

- увеличить CPU/RAM контейнера `data-service`;
- увеличить ресурсы PostgreSQL;
- увеличить ресурсы Redis, если очередь уведомлений станет большой;
- вынести Grafana/Loki отдельно, если логов станет много.

Горизонтальное масштабирование:

- запускать несколько экземпляров `api-gateway`;
- запускать несколько экземпляров `data-service`, потому что состояние лежит в PostgreSQL;
- запускать несколько worker-ов `task-service`, если очередь уведомлений растет;
- оставить `bot` в одном экземпляре, если MAX polling не поддерживает несколько конкурентных poller-ов без дублей.

Важное ограничение: состояние пошаговых сценариев бота сейчас находится в памяти процесса. Если запускать несколько экземпляров бота, нужно вынести это состояние в Redis или другую общую storage-систему.

## 22. Типовые проблемы

### Бот не отвечает

Проверить:

```bash
docker compose logs -f bot
```

Частые причины:

- не указан `MAX_BOT_TOKEN`;
- указан неполный токен;
- API gateway не healthy;
- бот не может достучаться до `http://api-gateway:3050`;
- пользователь пишет не тому боту.

### DataService не проходит healthcheck

Проверить:

```bash
docker compose ps
docker compose logs -f data-service
docker compose logs -f postgres
```

Частые причины:

- PostgreSQL еще не поднялся;
- неправильный `DATABASE_URL`;
- порт `5432` занят локальным PostgreSQL;
- локальный volume PostgreSQL содержит данные от другой схемы проекта.

Сброс локальной базы разработки:

```bash
docker compose down -v
docker compose up --build
```

### Отмененная запись все еще видна в "Моих записях"

В "Моих записях" должны показываться только активные записи. Если отмененная запись отображается:

- проверить, что бот и data-service пересобраны;
- проверить статус записи в `registrations`;
- проверить логи события отмены.

### Мероприятие не создается

Проверить:

- роль пользователя: нужен `organizer` или `admin`;
- привязку пользователя к вузу;
- корректность даты;
- корректность слотов;
- что слоты не пересекаются;
- логи `bot`, `api-gateway`, `data-service`.

### Не видно логов в Grafana

Проверить:

```bash
docker compose ps
docker compose logs -f fluent-bit
docker compose logs -f loki
```

Частые причины:

- Loki еще не ready;
- Fluent Bit не видит `/var/lib/docker/containers`;
- сервисы не пишут JSON-логи;
- выбран неправильный label в LogQL.

## 23. Глоссарий

- Админ вуза - администратор, ограниченный одним вузом.
- Организатор - пользователь, управляющий мероприятиями своего вуза.

- Слот - конкретный временной интервал мероприятия.
- Запись - факт регистрации пользователя на слот.
- Soft delete - удаление через флаг/дату удаления без физического удаления строки.
- API gateway - единая входная точка для клиентов.
- DataService - сервис данных и БД-логики.
- TaskService - сервис фоновых задач.
- Business log - лог бизнес-события.
- Technical log - технический лог сервиса.
- Bot JWT - короткоживущий асимметричный токен, которым бот подтверждает запрос к API gateway.
- Client JWT - симметричный токен внешнего клиента, выданный API gateway после проверки кода.
