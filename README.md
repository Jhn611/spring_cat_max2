# Весенний_код_40

Бот MAX для записи абитуриентов на мероприятия университета.

Реализовано:

- первое приветствие, дисклеймер хакатон-команды и согласие на обработку минимальных данных профиля;
- каталог мероприятий с датой, длительностью, форматом и свободными местами;
- подробная карточка мероприятия, выбор слота, подтверждение записи и код записи;
- отмена записи с учетом правила поздней отмены;
- уведомления по конкретной записи, которые пользователь может отключить;
- меню организатора: список закрепленных мероприятий, лимиты, список записей, поиск по коду, смена статусов;
- уведомления только активным участникам выбранного мероприятия.

## Быстрый старт

### Локально через Node

```bash
npm install
copy .env.example .env
npm run dev:service
```

Во втором терминале:

```bash
npm run dev
```

### Через Docker

```bash
copy .env.example .env
docker compose up --build
```

Или через npm-скрипт:

```bash
npm run docker:up
```

Остановить:

```bash
docker compose down
```

В `.env` нужно указать:

```env
MAX_BOT_TOKEN=полный_токен_бота
DATA_SERVICE_URL=http://localhost:3060
DATA_SERVICE_PORT=3060
DATABASE_URL=postgres://springcat:springcat@localhost:5432/springcat
TECH_ADMIN_MAX_IDS=ваш_max_id
ADMIN_MAX_IDS=ваш_max_id
LEGAL_DOC_VERSION=hackathon-2026-05-14
```

В Docker `DATA_SERVICE_URL` можно оставить как в примере: `docker-compose.yml` сам переопределит его для контейнера бота на `http://data-service:3060`.

Чтобы узнать свой MAX ID, запустите бота и отправьте `/whoami`.

## Команды

- `/start` - главное меню
- `/events` - каталог мероприятий
- `/my` - мои записи
- `/org` - меню организатора
- `/admin` - админ-панель технического админа
- `/admin_add MAX_ID` - назначить админа
- `/admin_org MAX_ID` - назначить организатора
- `/admin_remove MAX_ID` - снять права
- `/admin_user MAX_ID` - открыть карточку пользователя
- `/find КОД` - найти запись по коду в меню организатора
- `/whoami` - показать MAX ID пользователя

## Роли и админы

В боте есть четыре роли:

- `applicant` - обычный абитуриент, видит каталог и свои записи.
- `organizer` - организатор, видит закрепленные мероприятия и записи по ним.
- `admin` - админ, видит и управляет всеми мероприятиями.
- `tech_admin` - главный/технический админ, может назначать и снимать админов прямо в MAX.

Первого технического админа проще всего задать через `.env`:

```env
TECH_ADMIN_MAX_IDS=ваш_max_id
```

После этого техадмин открывает в MAX:

```text
/admin
```

И может назначать людей командами:

```text
/admin_add 123456789
/admin_org 123456789
/admin_remove 123456789
/admin_user 123456789
```

Роли сохраняются в Postgres. Если назначить роль пользователю, который еще не писал боту, сервис создаст запись-заглушку `MAX ID ...`; когда пользователь потом напишет боту, имя обновится, а роль сохранится.

## Настройка мероприятий

Демо-мероприятия находятся в `src/data-service/seed-events.ts`.

## Архитектура

Теперь бот и данные разделены:

- `src/bot` - модуль бота MAX. Он не работает с БД напрямую и обращается к `DATA_SERVICE_URL`.
- `src/data-service` - модуль сервиса данных: HTTP API, Postgres-слой и сиды мероприятий.
- `src/shared` - общий контракт: типы и доменные утилиты без зависимости от бота или БД.
- `Dockerfile` - собирает общий production-образ для обоих runtime-модулей.
- `docker-compose.yml` - запускает три контейнера: `postgres`, `data-service` и `bot`.

Сервис отдает мероприятия, свободные места, пользователей, записи, поиск по коду и обновление статусов. Бот занимается только диалогами MAX и вызывает сервис.

Такую структуру можно масштабировать вертикально и организационно: усиливать сервер сервиса данных отдельно от бота, выносить Postgres на отдельный сервер или managed DB, а контракт в `src/shared` оставить стабильным.

## Логика по слоям

`src/bot`

- принимает события и callback-кнопки MAX;
- показывает каталог, карточки мероприятий, мои записи и меню организатора;
- не знает, как устроена БД;
- получает данные через `src/bot/data-service-client.ts`.

`src/data-service`

- поднимает HTTP API на `DATA_SERVICE_PORT`;
- ходит в Postgres по `DATABASE_URL`;
- создает таблицы пользователей, мероприятий и записей;
- сидит демо-мероприятия из `src/data-service/seed-events.ts`;
- считает свободные места и отдает записи/статусы боту.

`src/shared`

- хранит общие типы: пользователь, мероприятие, слот, запись, статус;
- хранит маленькие доменные функции вроде проверки активного статуса;
- не импортирует код бота, HTTP-сервера или БД.

## Docker-сервисы

`postgres`

- образ: `postgres:16-alpine`;
- порт наружу: `5432`;
- база: `springcat`;
- пользователь: `springcat`;
- данные сохраняются в Docker volume `springcat-postgres`.

`data-service`

- команда: `node dist/data-service/index.js`;
- порт наружу: `3060`;
- ждет healthy-состояния `postgres`;
- подключается к БД по внутреннему адресу Docker: `postgres://springcat:springcat@postgres:5432/springcat`;
- выполняет миграции и сидит демо-мероприятия при старте.

`bot`

- команда: `node dist/bot/index.js`;
- использует `MAX_BOT_TOKEN`;
- обращается к сервису данных по внутреннему адресу Docker: `http://data-service:3060`;
- наружные порты не открывает.

Для вертикального масштабирования можно переносить `data-service` или `postgres` на более мощную машину/контейнер с большим лимитом CPU/RAM. Для горизонтального масштабирования можно запускать несколько экземпляров `data-service`, потому что состояние лежит в Postgres, а не в локальном файле.
