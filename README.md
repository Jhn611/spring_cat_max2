# Весенний_код_40

Бот MAX для записи абитуриентов на мероприятия университета.

Реализовано:

- первое приветствие, дисклеймер хакатон-команды и согласие на обработку минимальных данных профиля;
- каталог мероприятий с датой, длительностью, форматом и свободными местами;
- подробная карточка мероприятия, выбор слота, подтверждение записи и код записи;
- отмена записи с учетом правила поздней отмены;
- уведомления по конкретной записи, которые пользователь может отключить;
- меню организатора: список закрепленных мероприятий, лимиты, список записей, поиск по коду, смена статусов;
- уведомления только активным участникам выбранного мероприятия.

## Быстрый старт

### Локально через Node

```bash
npm install
copy .env.example .env
npm run dev:service
```

Во втором терминале:

```bash
npm run dev
```

### Через Docker

```bash
copy .env.example .env
docker compose up --build
```

Или через npm-скрипт:

```bash
npm run docker:up
```

Остановить:

```bash
docker compose down
```

В `.env` нужно указать:

```env
MAX_BOT_TOKEN=полный_токен_бота
DATA_SERVICE_URL=http://localhost:3060
API_GATEWAY_URL=http://localhost:3050
TASK_SERVICE_URL=http://localhost:3070
DATA_SERVICE_PORT=3060
API_GATEWAY_PORT=3050
TASK_SERVICE_PORT=3070
DATABASE_URL=postgres://springcat:springcat@localhost:5432/springcat
REDIS_URL=redis://localhost:6379
LOG_LEVEL=info
TECH_ADMIN_MAX_IDS=ваш_max_id
ADMIN_MAX_IDS=ваш_max_id
ADMIN_UNIVERSITY_ID=rtu-mirea
LEGAL_DOC_VERSION=hackathon-2026-05-14
```

В Docker бот ходит в `api-gateway` по внутреннему адресу `http://api-gateway:3050`; gateway уже сам обращается к `data-service` и `task-service`.

Логи пишутся в stdout/stderr контейнеров в JSON-формате. Смотреть их можно так:

```bash
docker compose logs -f api-gateway
docker compose logs -f data-service
docker compose logs -f task-service
docker compose logs -f bot
```

Уровень логов задается через `LOG_LEVEL`: обычно `info`, для подробной диагностики `debug`, для почти полной тишины `error`.

Полноценный pipeline логирования включен в Docker Compose:

- сервисы пишут структурированный JSON в stdout;
- `fluent-bit` читает docker-логи, парсит JSON и отправляет записи в Loki;
- `loki` хранит логи;
- `grafana` автоматически получает Loki datasource и dashboard `SpringCat Logs`.

Адреса:

- Grafana: `http://localhost:3000`, логин/пароль по умолчанию `admin/admin`;
- Loki: `http://localhost:3100`.

У логов есть поле `log_type`:

- `technical` - старты сервисов, HTTP-запросы, ошибки, worker-события;
- `business` - события процесса: создание/изменение/удаление мероприятия, запись пользователя, отмена записи, смена роли, постановка уведомления в очередь.

Примеры LogQL:

```logql
{project="springcat"}
{project="springcat", log_type="business"} | json
{project="springcat", service="bot"} | json | event="registration_confirmed"
{project="springcat", service="data-service"} | json | event="event_created"
```

Чтобы узнать свой MAX ID, запустите бота и отправьте `/whoami`.

## Команды

- `/start` - главное меню
- `/events` - каталог мероприятий
- `/my` - мои активные записи на мероприятия
- `/org` - мероприятия организатора и управление ими
- `/org_create` - пошагово создать мероприятие своего вуза, включая дату `ДД.ММ.ГГГГ` и слоты с временем
- `/org_edit` - пошагово изменить мероприятие своего вуза
- `/admin` - админ-панель технического админа
- `/admin_add MAX_ID` - назначить админа
- `/admin_org MAX_ID` - назначить организатора
- `/admin_remove MAX_ID` - снять права
- `/admin_university MAX_ID university_id` - сменить вуз админа или организатора
- `/admin_user MAX_ID` - открыть карточку пользователя
- `/find КОД` - найти запись по коду в меню организатора
- `/whoami` - показать MAX ID пользователя

## Роли и админы

В боте есть четыре роли:

- `applicant` - обычный абитуриент, видит каталог и свои записи.
- `organizer` - организатор, видит закрепленные мероприятия и записи по ним.
- `admin` - админ конкретного вуза, видит и управляет мероприятиями своего вуза.
- `tech_admin` - главный/технический админ, может назначать админов и менять им вуз прямо в MAX.

Первого технического админа проще всего задать через `.env`:

```env
TECH_ADMIN_MAX_IDS=ваш_max_id
```

После этого техадмин открывает в MAX:

```text
/admin
```

И может назначать людей командами:

```text
/admin_add 123456789
/admin_org 123456789
/admin_remove 123456789
/admin_user 123456789
```

Для мультивузовой схемы у админа и организатора есть привязка к вузу:

```text
/admin_add 123456789 rtu-mirea
/admin_org 123456789 rtu-mirea
/admin_university 123456789 spring-demo
```

Техадмин может назначать админов и организаторов в любой вуз. Админ вуза может назначать только организаторов своего вуза. Роли и привязка к вузу сохраняются в Postgres. Если назначить роль пользователю, который еще не писал боту, сервис создаст запись-заглушку `MAX ID ...`; когда пользователь потом напишет боту, имя обновится, а роль и вуз сохранятся.

## Настройка мероприятий

Вузы, мероприятия и слоты хранятся в Postgres:

- `universities` - вузы;
- `events` - мероприятия;
- `event_slots` - слоты мероприятий.

TS-файлов с сидами больше нет. Если нужно поменять мероприятия, меняй строки в БД. Bootstrap-демо создается только как стартовые строки при миграции, чтобы новая база не была пустой.

Удаление мероприятий мягкое: событие помечается `deleted_at`, скрывается из публичного каталога, но записи участников остаются в БД. В меню организатора или админа своего вуза такое мероприятие можно восстановить.

Создание и изменение мероприятий в боте идут пошагово. Дата вводится в формате `ДД.ММ.ГГГГ`, но бот также нормально приводит ручной ввод вроде `5.6.2026` или `05/06/2026` к нужному виду. Время начала мероприятия берется из первого слота. Слоты можно создать кнопками: сначала выбирается час начала, потом минуты, а конец слота считается по длительности. Ручной ввод тоже поддерживается: `15:00-16:00, 17:00-18:00`. У каждого слота конец должен быть позже начала, а слоты не должны пересекаться.

На шагах выбора даты и слотов бот показывает кнопки. Дата выбирается через календарь: дни разложены по неделям, месяц можно переключать вперед в пределах 6 месяцев от текущего месяца. Время внутри слота выбирается в два шага: сначала час `00`-`23`, затем минуты с шагом 5 минут. На каждом шаге есть кнопка отмены.

Длинные списки в боте выводятся страницами по 3 карточки или строки за раз. При перелистывании бот удаляет старый экран и показывает следующую страницу, чтобы чат не засорялся.

## Архитектура

Теперь бот, входной API, данные и фоновые задачи разделены:

- `src/bot` - модуль бота MAX. Он не работает с БД напрямую и обращается к `API_GATEWAY_URL`.
- `src/api-gateway` - входной Fastify-шлюз для бота, будущей React-админки и других клиентов. Он скрывает внутренние адреса сервисов.
- `src/data-service` - модуль сервиса данных: Fastify API, Drizzle ORM-схема и Postgres-слой.
- `src/task-service` - сервис фоновых задач: Fastify API, BullMQ, Redis и worker уведомлений MAX.
- `src/shared` - общий контракт: типы и доменные утилиты без зависимости от бота или БД.
- `observability` - конфиги Fluent Bit, Loki и Grafana для централизованных логов.
- `Dockerfile` - собирает общий production-образ для runtime-модулей.
- `docker-compose.yml` - запускает `postgres`, `redis`, `data-service`, `task-service`, `api-gateway`, `bot`, `fluent-bit`, `loki` и `grafana`.

DataService отдает мероприятия, свободные места, пользователей, записи, поиск по коду и обновление статусов. TaskService выполняет фоновые уведомления через очередь. Бот занимается только диалогами MAX и вызывает gateway.

Такую структуру можно масштабировать вертикально и организационно: усиливать сервер сервиса данных отдельно от бота, выносить Postgres на отдельный сервер или managed DB, а контракт в `src/shared` оставить стабильным.

## Логика по слоям

`src/bot`

- принимает события и callback-кнопки MAX;
- показывает каталог, карточки мероприятий, мои записи и меню организатора;
- не знает, как устроена БД;
- получает данные и ставит задачи через `src/bot/data-service-client.ts`, который обращается к API gateway.

`src/api-gateway`

- поднимает входной HTTP API на `API_GATEWAY_PORT`;
- проксирует доменные запросы в `data-service`;
- проксирует фоновые задачи по `/tasks/*` в `task-service`;
- является будущей точкой входа для React + TS админки.

`src/data-service`

- поднимает HTTP API на `DATA_SERVICE_PORT`;
- ходит в Postgres по `DATABASE_URL`;
- содержит Drizzle ORM-схему таблиц в `src/data-service/schema.ts`;
- создает таблицы вузов, пользователей, мероприятий, слотов и записей;
- хранит привязку админов и организаторов к вузам;
- создает стартовые демо-строки прямо в Postgres при миграции;
- считает свободные места и отдает записи/статусы боту.

`src/task-service`

- поднимает HTTP API на `TASK_SERVICE_PORT`;
- подключается к Redis по `REDIS_URL`;
- принимает задачи уведомлений через `/notifications`;
- выполняет отправку MAX-сообщений worker-ом BullMQ с retry/backoff.

`src/shared`

- хранит общие типы: пользователь, мероприятие, слот, запись, статус;
- хранит маленькие доменные функции вроде проверки активного статуса;
- не импортирует код бота, HTTP-сервера или БД.

## Docker-сервисы

`postgres`

- образ: `postgres:16-alpine`;
- порт наружу: `5432`;
- база: `springcat`;
- пользователь: `springcat`;
- данные сохраняются в Docker volume `springcat-postgres`.

`data-service`

- команда: `node dist/data-service/index.js`;
- порт наружу: `3060`;
- ждет healthy-состояния `postgres`;
- подключается к БД по внутреннему адресу Docker: `postgres://springcat:springcat@postgres:5432/springcat`;
- выполняет миграции и создает стартовые демо-строки при старте.

`redis`

- образ: `redis:7-alpine`;
- порт наружу: `6379`;
- хранит очередь BullMQ для фоновых задач.

`task-service`

- команда: `node dist/task-service/index.js`;
- порт наружу: `3070`;
- ждет healthy-состояния `redis`;
- использует `MAX_BOT_TOKEN` для отправки уведомлений.

`api-gateway`

- команда: `node dist/api-gateway/index.js`;
- порт наружу: `3050`;
- ждет healthy-состояния `data-service` и `task-service`;
- наружный вход для бота и будущей админ-панели.

`bot`

- команда: `node dist/bot/index.js`;
- использует `MAX_BOT_TOKEN`;
- обращается к API gateway по внутреннему адресу Docker: `http://api-gateway:3050`;
- наружные порты не открывает.

`fluent-bit`

- читает docker JSON logs из `/var/lib/docker/containers`;
- разворачивает JSON, который пишут сервисы;
- отправляет записи в Loki с labels `project`, `environment`, `service`, `log_type`.

`loki`

- образ: `grafana/loki`;
- порт наружу: `3100`;
- хранит логи в Docker volume `springcat-loki`.

`grafana`

- образ: `grafana/grafana`;
- порт наружу: `3000`;
- datasource Loki и dashboard создаются из `observability/grafana/provisioning`.

Для вертикального масштабирования можно переносить `data-service` или `postgres` на более мощную машину/контейнер с большим лимитом CPU/RAM. Для горизонтального масштабирования можно запускать несколько экземпляров `data-service`, потому что состояние лежит в Postgres, а не в локальном файле.
